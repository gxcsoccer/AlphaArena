## 🔴 遇到的问题和解决方案

### 问题 1: PR 包含多个独立功能
**现象**: PR #290 最初同时包含了 Issue #286（通知系统）和 #287（策略比较工具）的代码

**解决方案**: 
- QA 在审查时指出问题
- Dev 将代码拆分为独立的 PR
- 每个 PR 只处理一个 Issue

**经验教训**: 一个 PR 只应处理一个 Issue，便于代码审查和问题追踪

---

### 问题 2: 前端使用模拟数据而非真实 API
**现象**: `useStrategyComparison` Hook 使用 `generateMockComparisonResult()` 而非调用后端 API

**解决方案**: 
- Dev 修改 Hook 调用真实的 `/api/strategies/compare` API
- 确保前后端真正集成

**经验教训**: 开发时应尽早验证前后端集成，避免使用模拟数据到最后一刻

---

### 问题 3: 测试模块解析失败
**现象**: 
- 通知模块测试无法找到模块路径
- Jest 不支持 `import.meta` 导致前端测试失败

**解决方案**: 
- 修复测试文件中的导入路径
- 更新 Jest 配置以支持 ESM 风格导入

**经验教训**: 测试应在开发过程中持续运行，而非等到最后才验证

---

### 问题 4: TypeScript 编译和 ESLint 错误
**现象**: 
- 未使用的导入
- 未使用的参数
- 类型定义问题

**解决方案**: 
- 清理未使用的导入和参数
- 修复类型定义问题

**经验教训**: 提交前应运行 lint 和类型检查

---

### 问题 5: 合并冲突
**现象**: PR #290 与 main 分支存在合并冲突

**解决方案**: 
- `git rebase origin/main` 解决冲突
- `git push --force-with-lease` 更新 PR

**经验教训**: 在开发过程中定期同步 main 分支，减少冲突

---

### 问题 6: Vercel 部署失败
**现象**: 部署失败与构建错误相关

**解决方案**: 修复编译错误后部署自动通过

**经验教训**: CI/CD 是代码质量的重要保障

## 💡 改进建议

### 1. 开发流程改进
- **原子 PR**: 每个 PR 只处理一个 Issue，避免功能混杂
- **持续集成**: 开发过程中持续运行测试和 lint
- **提前集成**: 尽早验证前后端集成，避免使用模拟数据

### 2. 代码质量改进
- **PR 检查清单**: 提交前检查 TypeScript 编译、ESLint、测试通过
- **测试覆盖**: 确保测试在 CI 环境中可运行
- **类型安全**: 避免 `as any` 等类型断言

### 3. 协作改进
- **明确分工**: Issue 和 PR 的对应关系要清晰
- **及时沟通**: 发现问题及时在 PR 中讨论
- **代码审查**: QA 审查发现了很多问题，证明审查流程的价值

### 4. 技术债务
以下是 QA 在审查中提出的改进建议（后续迭代可处理）：

**通知系统**:
1. 扩展 Express Request 类型替代 `(req as any)`
2. 移除冗余轮询，仅依赖 Realtime 订阅
3. 测试端点添加生产环境禁用检查
4. 实现 quiet hours 时区处理
5. 优化广播通知的 N+1 查询问题

**策略比较工具**:
1. API 端点返回的策略/交易对从配置或数据库获取，而非硬编码
2. 为 StrategyComparisonPage 等组件添加测试
